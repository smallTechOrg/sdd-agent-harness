# Roadmap

## What This Agent Does

A personal on-demand data analysis agent that lets you upload CSV or Excel files (Phase 1–2) or connect directly to a PostgreSQL database (Phase 2), ask plain-English questions about the data, and receive a text answer plus an interactive Plotly.js chart rendered inline in the chat UI. The agent generates Python/Pandas code or SQL locally, executes it locally, and sends only a small aggregated result sample to the LLM for reasoning — raw data never leaves the machine.

## Who Uses It

A single personal user (the owner of the machine). The user is a data practitioner or analyst who wants a fast, private, conversational interface over their own datasets without exposing raw data to a cloud API.

## Core Problem Being Solved

Running ad-hoc data analysis requires switching between notebooks, SQL clients, spreadsheet tools, and charting software. This agent collapses that into a single chat interface: upload a file or point at a database, ask a question in plain English, and get an answer with a chart — no code, no context-switching, no data leaving the local environment.

## Success Criteria

- [ ] A CSV file can be uploaded and its schema (column names, dtypes, 3-sample rows) is extracted and displayed before any question is asked.
- [ ] A plain-English question about an uploaded CSV produces a text answer and an interactive Plotly.js chart inline in the chat UI within 30 seconds on a standard laptop.
- [ ] The agent auto-selects the appropriate chart type (bar, line, scatter, pie) for each question based on the data shape.
- [ ] Pandas code generated by the agent executes locally; only a result sample of at most 500 rows is sent to the LLM.
- [ ] A PostgreSQL database can be connected via `POSTGRES_DSN` and queried with the same plain-English interface (Phase 2).
- [ ] Schema and file metadata persist in SQLite across server restarts so the user does not re-upload files between sessions (Phase 2).
- [ ] All analysis runs are recorded in SQLite and retrievable via `GET /api/analysis/runs`.

## What This Agent Does NOT Do (Out of Scope)

- No multi-user support, authentication, or authorization — single personal user only.
- No cloud storage or remote file upload — files are stored on the local filesystem.
- No LLM-generated code that performs file I/O, network calls, or `os` module operations (enforced by sandboxing).
- No automatic scheduling or recurring analysis.
- No export to PDF/Excel/Slack — results live in the UI only.
- No training or fine-tuning on past runs.
- No support for database engines other than PostgreSQL (Phase 2+) — MySQL, BigQuery, etc. are out of scope.
- No vector search or semantic similarity over data rows.

## Key Constraints

- Raw data rows must not be sent to the LLM; only aggregated results or samples of at most 500 rows may be included in LLM prompts.
- LLM provider is Gemini (`AGENT_GEMINI_API_KEY`); model is `gemini-2.5-flash` for planning nodes (low latency) and `gemini-2.5-pro` for reasoning nodes (higher quality) — both configurable via env vars.
- App metadata (uploaded file registry, run history, schema cache) lives in SQLite; PostgreSQL is the external data source only.
- The frontend is a Next.js static export served by FastAPI at `/app` (single-origin, port 8001). The run path is `cd frontend && pnpm build` followed by `uv run python -m src`.
- All LLM-generated Pandas code runs in a restricted `exec()` sandbox; forbidden imports (`os`, `sys`, `subprocess`, `open`, `socket`, `requests`, `urllib`) are blocked.
- LangSmith tracing must be active from Phase 1 (env vars `LANGCHAIN_TRACING_V2`, `LANGCHAIN_API_KEY`).

## Phases of Development

### Phase 1 — CSV Upload + Question → Answer + Chart

- **Goal:** Upload a CSV file, ask one plain-English question, and receive a text answer plus an interactive Plotly.js chart rendered inline in the chat UI. The agentic LangGraph graph is wired and running from day one. PostgreSQL integration and cross-session schema persistence are clearly labelled stubs in the UI.

- **Independent slices (parallel build units):**
  - `slice-db-schema` (backend) — Alembic migration adding `uploaded_files` and `analysis_runs` tables to SQLite. deps: none.
  - `slice-api-files` (backend) — `POST /api/files/upload` and `GET /api/files` endpoints; file storage; schema introspection (column names, dtypes, 3-row sample); register all `/api/*` routers in `src/api/__init__.py`. deps: slice-db-schema (needs migration to have run; declare as dependency).
  - `slice-agent-graph` (backend) — Replace skeleton `transform_text` node with `load_dataset`, `plan_analysis`, `execute_code`, `reason_answer`, `handle_error`, `finalize` nodes; extend `AgentState`; wire LangGraph graph; extend `run_agent()` to accept `file_id` + `question`; create `src/api/analysis.py` router (registered by `slice-api-files`'s `__init__.py`); structured logging via `structlog`; LangSmith env-var wiring. deps: slice-db-schema, slice-api-files (because `__init__.py` router registration lives in slice-api-files).
  - `slice-frontend` (frontend) — Replace skeleton page with: left sidebar (file list + drag-drop upload + "Connect PostgreSQL [Coming Phase 2]" stub badge); main chat panel (question input, answer history cards); each answer card shows text answer + Plotly.js chart; loading spinner; red error alert; schema preview modal after upload. deps: none (builds against documented API contract).

- **Key surfaces / files:**
  - `slice-db-schema`: `src/db/models.py` (add `UploadedFile`, `AnalysisRun` models), `alembic/versions/001_add_files_and_runs.py`
  - `slice-api-files`: `src/api/files.py`, `src/domain/file.py`, `src/api/__init__.py` (register router with `/api` prefix)
  - `slice-agent-graph`: `src/graph/state.py`, `src/graph/nodes.py`, `src/graph/edges.py`, `src/graph/agent.py`, `src/graph/runner.py`, `src/api/analysis.py`, `src/domain/analysis.py`, `src/prompts/plan_analysis.md`, `src/prompts/reason_answer.md`, `src/config/settings.py` (add `langchain_tracing_v2`, `langchain_api_key`, `postgres_dsn`, `llm_model_plan`, `llm_model_reason` fields), `tests/test_phase1_agent.py`

### Phase 2 — PostgreSQL Integration + Session Persistence

- **Goal:** Connect to an existing PostgreSQL database via `POSTGRES_DSN`, ask questions that generate SQL (executed locally), and have uploaded file metadata and schema persist in SQLite across server restarts. The "Connect PostgreSQL" stub in the sidebar becomes real.

- **Independent slices (parallel build units):**
  - `slice-pg-backend` (backend) — `POST /api/pg/connect` endpoint; `pg_connections` SQLite table; `load_dataset` node extended to handle `source_type=postgres`; SQL code-gen path in `plan_analysis`; schema introspection for PostgreSQL tables. deps: none.
  - `slice-session-persist` (backend) — On file upload, persist schema_json in `uploaded_files`; on server start, reload file list from SQLite so the sidebar is populated without re-uploading. deps: none.
  - `slice-pg-frontend` (frontend) — Wire the "Connect PostgreSQL" sidebar form (DSN input + test connection button); show connected DB tables alongside uploaded files; display SQL (not Pandas code) in the answer card when source is PostgreSQL. deps: none.

- **Key surfaces / files:**
  - `slice-pg-backend`: `src/db/models.py` (add `PgConnection`), `alembic/versions/002_add_pg_connections.py`, `src/api/pg.py`, `src/graph/nodes.py` (extend `load_dataset`, `plan_analysis`)
  - `slice-session-persist`: `src/graph/runner.py`, `src/api/files.py` (restore schema from DB on startup)
  - `slice-pg-frontend`: `frontend/src/components/Sidebar.tsx` (wire PG form), `frontend/src/components/PgConnectForm.tsx`

### Phase 3 — Excel Upload + Multi-Turn Follow-Ups + Full Resilience

- **Goal:** Add Excel file support (`.xlsx`, `.xls`), enable multi-turn follow-up questions that reference previous results in the same chat session without re-specifying the dataset, and harden all error paths (malformed files, LLM code-gen failures, unreachable PostgreSQL).

- **Independent slices (parallel build units):**
  - `slice-excel` (backend) — Extend `load_dataset` to parse Excel via `openpyxl`; accept `.xlsx`/`.xls` in the upload endpoint. deps: none.
  - `slice-multiturn` (backend) — Extend `AgentState` with `conversation_history` (list of prior question/answer/chart_spec turns); pass conversation history into `plan_analysis` and `reason_answer` LLM prompts so follow-up questions resolve naturally; store full turn history in `analysis_runs`. deps: none.
  - `slice-resilience` (backend) — Wrap all LLM calls and `exec()` sandbox in retry logic (max 2 retries on transient errors); timeout on `execute_code` (30 s); graceful `handle_error` with user-facing messages; PostgreSQL connection retry with back-off. deps: none.
  - `slice-phase3-frontend` (frontend) — Multi-turn chat history rendered per-session; follow-up question input stays pre-populated with the active dataset; Excel file badge in the sidebar. deps: none.

- **Key surfaces / files:**
  - `slice-excel`: `src/graph/nodes.py` (`load_dataset`), `src/api/files.py`, `pyproject.toml` (add `openpyxl`)
  - `slice-multiturn`: `src/graph/state.py`, `src/graph/nodes.py`, `src/prompts/plan_analysis.md`, `src/prompts/reason_answer.md`, `src/db/models.py` (`AnalysisRun.conversation_history_json`)
  - `slice-resilience`: `src/graph/nodes.py`, `src/graph/runner.py`, `src/llm/client.py`
  - `slice-phase3-frontend`: `frontend/src/components/ChatPanel.tsx`, `frontend/src/components/AnswerCard.tsx`
