# UI

## UI Type

Web browser — a **single-page shell** (`index.html`) server-rendered via Jinja2 with progressive-
enhancement JavaScript (no framework). Three routes render the **same** shell with different initial
state, so deep links and the no-JS path both work and the persistent chrome (header tabs + sidebar)
stays put:

- `GET /` — shell, no active entity (Analyse tab).
- `GET /sessions/{id}` — shell with that session active and its conversation thread server-rendered
  (Analyse tab).
- `GET /database/{id}` — shell with that server active (Database tab).

Switching the active session or server is plain navigation to those routes (the route's `active_tab`
keeps you on the right tab). JS handles tab toggling, the token-usage widget, the upload→create two-step,
the ask overlay + status polling, inline confirms, **loading overlays** (a full-page overlay for
delete/sync and for **navigating into a session/database** — those routes re-render server-side, so the
overlay covers the gap; a section-scoped overlay over one capability list during its delete), and
**"Load more" pagination** (each list renders its first `ui_page_size` rows — default **5** — inline + a
button that appends the next window as an HTML fragment, rendered with the same row macros, so a paged-in
row is identical to a server-rendered one). All mutations remain form POSTs that redirect; the shell is
served `Cache-Control: no-store`, so a post-delete redirect always shows a freshly-rendered list (never a
cached one still listing the just-deleted row).

## Persistent chrome

**Header (sticky, blue `#1e40af`):** brand "DataAnalyst AI" (links home) · centre **Analyse / Database**
tab buttons · right-side current LLM model name. A **stub-mode banner** sits above it when
`llm_provider == "stub"`.

**Sidebar (left):**
- **Sessions** list with a **+ New** button (opens the new-session modal). Each row: name, question
  count + date, a chip per attached server, and a delete (✕, confirm → full-page overlay). The active
  session is highlighted. Paginated newest-first with a **Load more** button.
- **Token-usage widget** (bottom): model name; **Last query** (in/out/cost) and **Today**
  (in/out/queries/cost); storage (server count, total rows). Fed by `GET /stats/daily`; cost is the
  server-computed `estimated_cost_usd` (no client-side pricing).

## Tabs

### Analyse tab (`active_tab="analyse"`)

- **Top row — two cards:**
  - **Databases:** total count, then a row per server — name (→ Database tab), type + `v{version}`
    pills, "{n} tables · {rows} rows · {t} tools · {r} resources · {p} prompts", credential-free URI
    (`DatasetURI.display()`), and **Inspect / Sync / Delete** actions (Sync/Delete raise the full-page
    overlay). `connection_error` shows an error pill. Empty state when none; paginated newest-first with
    a **Load more** button.
  - **Upload data files:** a database-name field + a **Source** selector for the database type. For
    `parquet`, a **drag-and-drop / click** drop zone (.csv .tsv .json .xlsx .xls); **Create database**
    runs the two-step the UI performs — `POST /database/upload` (file → Parquet, returns the data URI)
    then `POST /database` (creates + auto-syncs). For an external type (postgresql / sqlite / mongodb /
    snowflake) a **connection-URI** field is shown — its placeholder shows **only the selected type's**
    URI shape (each `DATABASE_TYPES` entry carries its own `uri_hint`) — and `POST /database` is called
    directly.
- **Conversation card (fills remaining height):**
  - With an active session: header (session name + server chips), the **thread**, and the **ask** form
    (textarea + **Ask**) pinned at the bottom. The thread is the scroll area, **chronological with the
    newest turn at the bottom** (chat-style) and starts pinned there; older turns load on demand (a
    **Load older** button that also auto-fires on scroll-to-top) and prepend with stable scroll. A turn
    shows the question, the answer, meta pills (iterations · tokens · cost · time), and a collapsible
    **agent-reasoning trace** (each step's `tool`, optional `capability`, SQL, and result). An overlay
    with cycling step labels covers the card while a query runs; `?new=` polls
    `GET /sessions/{id}/query/{qr}/status` until done.
  - With no active session: a placeholder prompting to pick a session or create one.

### Database tab (`active_tab="database"`)

Surfaces a server's MCP architecture (per the spec's three child kinds):

- With an active server: a header bar (name, type + `v{version}` pills, title, credential-free URI +
  last-sync, **Sync** button, **+ Add table** for parquet → upload + sync), then a two-pane body:
  - **EER diagram** (left): a colour-type legend (number / text / date / other) and a **table box per
    entity** (columns with a type-coloured dot, name, dtype, and an **FK** badge on foreign-key columns),
    with **edges drawn between foreign-key relationships** (from the schema resource). Rendered client-side
    **entirely from the resources table** (entity columns + the canonical `{from_table, from_column,
    to_table, to_column}` relationships) — the connector is never called to view it.
  - **MCP capabilities** (right), the active rows only, each manually editable via the JSON-RPC write
    surface (a modal → `POST /database/{id}`, then reload). Each list shows its **total** count, renders
    the first page, and **Load more**s the rest; a **Delete** covers only that section with a spinner
    while it runs (Edit is atomic — no overlay):
    - **Tools** — name, description, the `execution.sql_template`, and parameter pills; **+ Add tool**,
      per-tool **Edit** (name, description, sql_template, parameters JSON) and **Delete** (hard).
    - **Resources** — kind pill + uri + description + **size**. Entity resources: **Edit** (title,
      description, and **entity kind** — flip `primary_entity` ↔ `secondary_entity`) and **Delete**
      (drops the backing table). The schema (entity-relationship) resource: **Edit relationships** only —
      no delete, no re-kind (it is tied to the server).
    - **Prompts** — name + description; **+ Add prompt**, per-prompt **Edit** and **Delete** (hard).
- With no active server: a placeholder + a quick-pick list of servers.

## Modals

- **New session:** optional name + a checkbox list of MCP servers (field `database_ids`, ≥1 required) →
  `POST /sessions`. When there are no servers, it explains to upload a dataset first.
- Destructive actions (delete server / session) use a native confirm.

## Error States

- Pipeline errors: the failed turn renders inline (`⚠ Failed: …`) — never a bare 500.
- Upload/create errors: a credential-free message under the Upload card.
- 404: structured `{code, message}` (the API error envelope).

## Tech Stack

Jinja2 + FastAPI, one shared `index.html` over `base.html`. Inline CSS in the demo's blue design
language; vanilla JS for tabs, the token widget, upload→create, the ask overlay/poll, and confirms.
Stub-mode banner via the `llm_provider` context variable; model name via `llm_model`.
