# Capability 4: Generate & Serve MCP Capabilities (Sync Pipeline + JSON-RPC Surface)

## Overview

Every MCP server's **tools, resources, and prompts** are LLM-generated by a **5-stage sync pipeline** and
served over a standards-compliant **JSON-RPC endpoint** (`POST /database/{id}`). Sync is **incremental**
(the LLM is shown the existing capabilities), **versioned** (each apply bumps `Database.version`), and
**soft-delete-only** (a suggestion can never irreversibly delete a capability).

## The Sync Pipeline (`tools/sync/`)

`run_sync(database_id) -> SyncResult`. Each stage builds a prompt carrying (a) the **first 5 rows** of each
physical table and (b) the server's **existing** capabilities (so the LLM edits incrementally), calls the
LLM, parses JSON, and **falls back to a deterministic template on any failure** (sync never hard-fails).
Each stage injects a distinct node tag so the stub branches deterministically (see `ai-agents.md` §8).

| Stage | Tag | Produces | Persists to |
|------|-----|----------|-------------|
| a. title + description | `<node:sync_title>` | `{title, description}` | `databases.title`, `.description` |
| b. entity-relationship schema | `<node:sync_schema>` | `{relationships:[{from_table,from_column,to_table,to_column}]}` (from the connector's FKs when available, else the LLM) | the `kind='schema'` resource's `content` (the schema source of truth) |
| c. entities → resources | `<node:sync_entities>` | `{entities:[{name,title,description,kind,uri,mime_type}]}` | `mcp_resources` (primary/secondary) |
| d. GET-APIs → tools | `<node:sync_tools>` | `{tools:[{name,title,description,output_schema?,execution:{sql_template,parameters:[{name,type,description?}]}}]}` | `mcp_tools` |
| e. prompts | `<node:sync_prompts>` | `{prompts:[{name,title,description,arguments,template}]}` | `mcp_prompts` |

**Tool `execution` object.** A tool is stored as a single `execution_json` object — `{"sql_template":
"SELECT … WHERE x > $min", "parameters": [{"name","type","description?","required?"}]}` — the source of
truth for the executor. `sql_template` carries the `$param` embeddings; `parameters` declares each
embedding's type. There is **no separate `sql_template` or `input_schema` column**: the bare
`sql_template` and the MCP-advertised `inputSchema` are *derived* (read-only) from `execution`.

**Schema vs entity resources (the dataset's schema source of truth → EER diagram).** The `kind='schema'`
**entity-relationship** resource holds **foreign-key relationships only**, in ONE canonical format
regardless of database type and what the LLM must emit:
`{relationships: [{from_table, from_column, to_table, to_column}]}`. Relationships come from the
connector's FK inspection when it exposes them (`discover_relationships` — Postgres `information_schema`,
SQLite `PRAGMA foreign_key_list`), otherwise the LLM stage backfills the same shape. Each **entity**
resource carries its backing table's columns (`content = {table, columns, row_count}`). Together they
fully describe the schema — the **Database tab renders the EER diagram (table boxes + FK edges) entirely
from these resources, never re-calling the connector**. Every resource also stores a **`size`** (bytes).
A resource **`uri` is unique at the DB level** (partial-unique over active rows, so a pruned entity can
be re-added).

**Inspection happens only in sync.** `discover_tables`/`discover_relationships` (which hit the underlying
store) run **only** during the sync pipeline, which materializes the result into the resources table.
Everywhere else — the agent pool, `tools/call`, write-time validation, the EER view — reads the resources
table (`active_entity_tables`) and the connector merely **builds query views over the given table names**
(`build_server(table_names)`); the serving path never re-inspects the store.

The cascade threads b → c → d → e: the schema feeds the entities; entities feed the tools; tools feed the
prompts. (This is the dependency chain that the Phase-B partial syncs follow in reverse.)

### Apply (transactional, versioned, soft-delete-only)

In **one transaction**:
1. Load the server's **active** capabilities (per type).
2. `new_version = server.version + 1`.
3. For each proposed item, match an existing active row by stable key (`name` for tools/prompts;
   `uri` or `kind`+name for resources): match → **update in place**; new → **insert** with
   `created_version = new_version`.
4. Any active row **absent** from the proposal → set `deleted_at` + `deleted_version = new_version`
   (**never** hard-delete).
5. **Validate** each generated `sql_template` compiles (run it with `LIMIT 0`); skip/flag a tool whose
   SQL doesn't compile rather than persisting a broken tool.
6. Set `server.version = new_version`, title/description, `last_synced_at`,
   `last_sync_status` (`ok`/`partial`/`error`). Commit (rollback on any error → no half-applied sync).
7. Close the pools of sessions attached to this server (lock-safe).

### Stub determinism (offline)

The stub provider has one branch per stage tag. Each echoes identifiers parsed from its prompt (table
names → entities → tool names → prompt names) so the cascade is self-consistent, and emits a valid
**zero-parameter** SELECT (e.g. `SELECT COUNT(*) AS n FROM {table}`) for tools so `tools/call` and the
agent both work offline. A test asserts every stage tag is handled (a missing branch yields a garbage
sync).

## The JSON-RPC Surface (`POST /database/{id}`)

JSON-RPC 2.0 over the server's **active** capability rows (`tools/mcp/dispatch.py` — a DB-backed
dispatcher, **not** FastMCP). Read methods (Phase A):

- `tools/list {cursor?}` → MCP Tool objects + `nextCursor?`
- `tools/call {name, arguments}` → bind `arguments` into `sql_template` (DuckDB **parameter binding**),
  run via the read-only guard → `CallToolResult {content, structuredContent?, isError}`. SQL failure →
  `isError:true`; unknown tool → JSON-RPC `-32602`.
- `resources/list {cursor?}` / `resources/read {uri}` → `{contents:[{uri,mimeType,text}]}` (`kind='schema'`
  returns the `kind='schema'` resource's relationship content).
- `prompts/list {cursor?}` / `prompts/get {name, arguments?}` → `{description, messages}`.
- Unknown method → `-32601`; bad params/cursor → `-32602`.

**Pagination:** opaque keyset cursor (base64 of the last `(created_at, id)`); `nextCursor` omitted on the
last page; page size `mcp_list_page_size` (default 50).

## Phase B — Granular mutation methods (implemented)

Six client-driven JSON-RPC mutation methods on `POST /database/{id}`. The client supplies a
stage-shaped capability `definition` in `params`; the method applies it, then runs a **partial sync** that
re-asks the LLM for the downstream capabilities. Params: `{"definition": {...}}`. Result:
`{ok, version, applied:{child,op,key}, cascade:{tools,prompts}, status}`.

| Method | Effect | Cascade |
|--------|--------|---------|
| `tools/add` | Insert a tool (with `execution`) | additive → regenerate **prompts** |
| `tools/update` | Update a tool (by `name`) | additive → regenerate **prompts** |
| `tools/delete` | **Hard-delete** a tool (by `name`) | prune → regenerate **prompts** |
| `prompts/add` | Insert a prompt | (none) |
| `prompts/update` | Update a prompt (by `name`) | (none) |
| `prompts/delete` | **Hard-delete** a prompt (by `name`) | (none — a leaf) |
| `resources/add` | Insert a resource (entity) | additive → regenerate **tools**, then **prompts** |
| `resources/update` | Update a resource (by `uri`) | see rules below |
| `resources/delete` | **Hard-delete** an entity (by `uri`): drop its backing table + clean relationships | prune → regenerate **tools**, then **prompts** |

### Manual delete (hard) + LLM-driven prune cascade

`*/delete` is a **manual** action, so the targeted row is **hard-deleted** (removed, not tomb­stoned —
unlike the reversible soft-delete the LLM sync uses). Its **cascade** re-runs the dependent LLM stages
and **prunes** (soft-deletes) dependents that no longer apply (an LLM-driven regen stays reversible).
`resources/delete` of an entity also **drops the backing physical table** (Parquet file + catalog entry)
and removes any relationships referencing it from the entity-relationship schema. One version bump.

### Resource update / schema rules

- An **entity** resource update edits **only `title` + `description`** (cosmetic — no cascade); its
  backing table/columns change only by adding/deleting tables.
- The **entity-relationship (schema) resource cannot be added or deleted** (it is tied to the server's
  existence). It **can be updated** manually — its canonical `relationships` list — which **cascades**
  (regenerate tools, then prompts, additively).

Rules (differ deliberately from the full `/sync`):
- **Two-tier deletion semantics.** The full `/sync` regenerates everything and **soft-deletes** dropped
  capabilities. A granular mutation cascade is **ADDITIVE**: it inserts new + updates matched downstream
  rows and **never** soft-deletes existing ones (least astonishment — a single add must not silently drop
  a sibling). The full `/sync` remains the only pruning operation.
- **Single version bump.** The explicit mutation + its whole cascade are one transaction and bump
  `version` exactly once; all rows written carry that `created_version`.
- **`add` rejects an active name/uri; `update` requires one.** A duplicate-active key → `-32602` (never an
  IntegrityError); an unknown key on update → `-32602`. A name whose only match is a tombstone re-inserts
  a fresh active row (partial-unique is over active rows).
- **Write-time `sql_template` guard** (`tools/add`/`update`): the SQL is `_guard_select`-checked
  (single-statement read-only SELECT/WITH; no `ATTACH`/`COPY`/`PRAGMA`/`INSTALL`/`LOAD`; no file-reading
  table functions like `read_text`/`read_csv`/`read_parquet`/`glob` — these would otherwise read arbitrary
  host files); zero-param SQL is compile-checked (`LIMIT 0`); every `$param` in the SQL must be declared in
  `input_schema.properties`. A bad template → `-32602`, never persisted. The same guard runs at execution
  (agent free-SQL, `tools/call`, generated tools), so a query can only touch the dataset's views.
- **Transaction + pool refresh.** A failed mutation rolls back (nothing persists); a successful one
  commits, then closes the pools of sessions attached to the server (`close_sessions_for_server`,
  post-commit) so the agent re-reads the new capabilities.
- `resources/add` of an entity not backed by a physical table cascades **no new tools** (tool generation
  is grounded in physical tables); adding a CSV (a new physical table) is the path that grows tools.
- Method names are exactly as above; any other method → `-32601`.

## Phase B — Hybrid agent consumption (implemented)

The session pool surfaces each server's active generated GET-API tools alongside the generic `query`
tool. **Addressing stays single-level with an optional `capability`:**
`{"tool":"<server>","arguments":{"query":"SELECT…"}}` runs free SQL (Phase A, unchanged);
`{"tool":"<server>","capability":"<gen tool>","arguments":{...params}}` runs that generated tool. Absent
`capability` ⇒ free SQL (so a generated tool literally named `query` does not collide). Generated tools
execute via the shared read-only path (`_run_select_params` on the server's DuckDB connection — same guard
as the dispatcher), so there is no second, unguarded SQL route. The agent prompt lists, per server, the
generic SQL option and any pre-built tools (name + params + description) and is told to prefer a matching
pre-built tool, else write SQL. The offline stub keeps emitting free SQL (the golden path is unchanged);
the hybrid routing is covered by pool/execution unit tests. The trace (`query_history_json`) carries an
optional `capability` only on generated-tool calls (free-SQL entries are byte-identical to Phase A).

## Success Criteria

- Creating a server runs sync and yields ≥1 tool, ≥1 resource (the schema), ≥1 prompt — offline (stub).
- Re-syncing (full `/sync`) bumps the version, updates matched, and **soft-deletes** dropped ones; a
  re-added name inserts a fresh active row without colliding with the tombstone.
- `POST /database/{id}` answers all six read methods with correct shapes, cursor pagination, and the
  `-32601`/`-32602`/`isError` error mapping.
- The six mutation methods: `add`/`update` enforce the active-key rules (`-32602`), bump version once,
  cascade **additively** (no sibling soft-deleted), reject a bad `sql_template` (`-32602`), persist nothing
  on failure, and refresh pools on success. An unimplemented method (e.g. `tools/delete`) → `-32601`.
- Hybrid: the agent can call a generated tool by `capability` (params bound, read-only-guarded) and still
  run free SQL when `capability` is absent; the golden path is unchanged.
