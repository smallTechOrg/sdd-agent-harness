# Data Model

> **Clean rebuild.** This is a v0.1 with no data to preserve. The schema is created by **one fresh
> initial Alembic migration** (`down_revision=None`). There is no backfill, no deprecated columns, and no
> compatibility aliases. The prior `data_sources` / `dataset_tables` model is fully replaced.

## Storage Technology

**PostgreSQL** (the master metadata store) via SQLAlchemy 2.0 (sync) stores all **application metadata**
(MCP servers + their capability rows, sessions, query history). The backend is configurable via
`DATAANALYSIS_DATABASE_URL` and SQLite is fully supported (it is the default and what the tests use), but
the deployed master database is **PostgreSQL** (`postgresql+psycopg2://…`). Partial-unique indexes carry
both `sqlite_where` and `postgresql_where` so soft-delete uniqueness holds on either backend. Alembic
manages schema migrations (batch mode is applied for SQLite only). For an **internal** (`parquet`) server
the **data itself** lives in **Parquet** files on disk and is queried by DuckDB — it is never copied into
the metadata DB. For an **external** (`postgresql`, BETA) *dataset* the data stays in the remote database;
only connection metadata and an introspected table catalog are persisted.

A database's **tools/resources/prompts ARE stored as rows** (unlike the prior model). They are
LLM-generated by the sync pipeline and must be individually addressable, versioned, soft-deletable, and
cursor-paginated — so they are typed child tables, not a JSON blob. There is **no physical-table catalog
column**: the connector inspects tables live (sync only), and the result is materialized into the **entity
resources** (per-table columns) + the **`kind='schema'` resource** (FK relationships — the source of
truth). No `physical_tables_json` and no `dataset_schema_json` column.

> **Reframe (this iteration).** An **MCP server** is a *type* (parquet, postgres, …) with its own
> connector logic in code (`tools/connectors/`). A **database** is an *instance* of that type — the
> primary entity (`databases` table). Each database is served as an MCP server of its type.

### Agent memory store (separate, not Alembic-managed)

Durable per-session memory is stored by LangGraph's `SqliteSaver` in its **own SQLite file** (path from
`DATAANALYSIS_CHECKPOINT_DB`), keyed by `thread_id = session_id`. Its tables are owned by
`langgraph-checkpoint-sqlite` — **not** part of the SQLAlchemy models or Alembic migrations.

## Entities

### Entity: Database (`databases` / `DatabaseRow`)

The primary entity — a dataset wrapped 1:1 by an MCP server. `name` is the dataset/server name (also the
agent "tool" name); `type` is the dataset type; `uri` is the dataset address (unique → 1:1). All other
dataset info lives in metadata columns.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key |
| name | TEXT | yes | Server/dataset name = agent tool name. **UNIQUE** |
| title | TEXT | no | LLM-generated display title (sync stage a) |
| description | TEXT | no | LLM-generated server description (sync stage a); used as the MCP server description + the agent tool description |
| type | TEXT | yes | The **MCP server type** (its connector): `parquet` (internal) · `postgresql` · `sqlite` · `mongodb` · `snowflake` (external; the last three BETA) |
| uri | TEXT | yes | Dataset URI. Internal: `parquet:///{name}`. External: `postgresql://user:pass@host:port/db` (credentials stored **plaintext** for v0.1 BETA; never logged/displayed — always via `DatasetURI.display()`). **UNIQUE** (1:1 dataset↔server) |
| version | INTEGER | yes | Monotonic capability version; default 1, bumped on each sync |
| last_synced_at | TIMESTAMP | no | When the sync pipeline last ran |
| last_sync_status | TEXT | no | `ok` / `partial` / `error` (for the UI) |
| connection_error | TEXT | no | Last sanitized connection-check error (never credentials); null when healthy |
| created_at | TIMESTAMP | yes | When the server was created |

> There is **no physical-table catalog column** — the connector inspects tables live (sync only) and the
> result is materialized into the entity resources (columns) + schema resource (FK relationships). Parquet
> files live at `{datasets_dir}/{slug(name)}/{table}.parquet`; the connector derives each path from the
> table name, so the URI carries no filesystem path.

### The capability children — shared envelope

Each of `mcp_tools`, `mcp_resources`, `mcp_prompts` carries the same versioning + soft-delete envelope:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key |
| database_id | TEXT (FK → databases.id) | yes | Owning server (indexed) |
| created_version | INTEGER | yes | `server.version` when this row was created |
| deleted_at | TIMESTAMP | no | Soft-delete tombstone — **NEVER hard-deleted**; null = active |
| deleted_version | INTEGER | no | `server.version` at soft-delete |
| created_at | TIMESTAMP | yes | Row creation time |
| updated_at | TIMESTAMP | yes | Last update time |

**Active view** (used everywhere a capability is listed/called): `database_id = ? AND deleted_at IS NULL
ORDER BY created_at, id`. Centralized in a repository helper so no endpoint forgets the filter.

### Entity: Session (`sessions` / `SessionRow`)

A named conversation over one or more MCP servers. The agent attaches one in-process pool server per
attached MCP-server entity. Joins **within** a server's tables are supported; cross-server joins happen
across ReAct iterations.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key |
| name | TEXT | no | Optional; defaults to "Session YYYY-MM-DD HH:MM" |
| created_at | TIMESTAMP | yes | Session start time |
| updated_at | TIMESTAMP | yes | Last activity time |

### Entity: SessionDatabase (`session_databases` join table)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| session_id | TEXT (FK → sessions.id) | yes | Composite PK |

## Relationships

```
Database (M) ──< SessionDatabase >── (N) Session
Database (1) ──< McpTool (N)        (active = deleted_at IS NULL)
Database (1) ──< McpResource (N)    (active = deleted_at IS NULL)
Database (1) ──< McpPrompt (N)      (active = deleted_at IS NULL)
Session   (1) ──< QueryRecord (N)
QueryRecord (1) ──< AgentRun (N)
```

## Bootstrap: Upload → Create → Sync

1. **Upload** (`POST /database/upload`, auxiliary): `database_name` + CSV → Parquet at
   `{datasets_dir}/{slug(name)}/{table}.parquet`; returns the data URI `parquet:///{name}`. No entity.
2. **Create** (`POST /database`): URI + type → reject duplicate URI/name (1:1) →
   **`connection_check()` via the connector before commit** → insert `Database` (version 1) →
   **`run_sync()`** inspects tables and populates tools/resources/prompts. A broken server is never persisted.
3. **Sync** (`POST /database/{id}/sync`): regenerate capabilities incrementally; **transactional apply**
   (insert/update/soft-delete) and `version++`.

## Versioning & Soft-Delete (non-negotiable)

- `Database.version` is a monotonic counter bumped once per sync apply.
- Sync (LLM-driven) **never hard-deletes** a capability — a removal sets `deleted_at` + `deleted_version`.
  Re-adding a previously-removed name inserts a new active row (the partial-unique index is over active
  rows only, so it doesn't collide with the tombstone). The **manual** `tools|prompts|resources/delete`
  methods DO hard-delete (a deliberate user action), then run a pruning cascade.
- The whole sync apply runs in **one transaction** (request session commit/rollback): a partial sync
  never persists a bumped version with half-updated children.

## Data Lifecycle

- **Database**: created on `POST /database`; its capabilities regenerated by sync; deleted via explicit
  user action — `shutil.rmtree` of the dataset directory + the server row (cascade to capability rows,
  `SessionDatabase` links, and QueryRecord/AgentRun via the session).
- **Capability rows**: created/updated/soft-deleted by sync; physically removed only when their server is
  deleted.
- **MCP pool**: per session — built lazily on the first query (`SessionPoolManager`), reused, idle/LRU
  evicted, closed on session delete + app shutdown. Built from each database's **resource tables**
  (`active_entity_tables`) + `uri` — never by re-inspecting the store. `close(session_id)` is lock-safe.
- **Session / QueryRecord / AgentRun**: standard create-on-use, delete-with-parent.

## Migration note

**Fresh, single initial migration** (`down_revision=None`, batch mode for SQLite): creates `databases`,
`mcp_tools`, `mcp_resources`, `mcp_prompts`, `sessions`, `session_databases`, `query_records`,
`agent_runs`, with `database_id` indexes and the three **partial-unique** indexes (`WHERE deleted_at IS
NULL`). No backfill. `downgrade` drops everything. `alembic upgrade head` from empty builds the schema;
`alembic downgrade base` → `upgrade head` round-trips.

## Sensitive Data

- Uploaded files may contain PII; Parquet files are stored as-is on local disk (no special handling in
  v0.1). Multi-user deployments must add access controls.
- No LLM API keys stored in the database.
- External (`postgresql`, BETA) credentials are stored **plaintext** in `databases.uri` for v0.1 — never
  logged or displayed (every rendering goes through `DatasetURI.display()`, which strips credentials). A
  read-only `ATTACH` lets the agent read any table the credentials can reach, so the DB user must be
  scoped. External datasets are gated by `DATAANALYSIS_ENABLE_EXTERNAL_DATASETS` (default off; create
  returns 501 when off).
