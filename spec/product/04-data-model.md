# Data Model

## Storage Technology

**SQLite** via SQLAlchemy 2.0 (sync) stores all **application metadata** (datasets, sessions, query history). Alembic manages schema migrations. For an **internal** (`parquet`) dataset the **data itself** lives in **Parquet** files on disk and is queried by DuckDB inside the dataset's MCP server — it is never copied into SQLite. For an **external** (`postgresql`, BETA) dataset the data stays in the remote database; only connection metadata and introspected schema are persisted.

A dataset's tools are **not** stored as rows. They are exposed at runtime by the dataset's MCP server and discovered via the MCP client (`list_tools`). The only tool-related data persisted is the LLM-generated **descriptions**: the dataset-level `tool_description` (on the dataset row) and the per-table `capability_description` (on each table row), so the server's tool and capability descriptions survive without re-calling the LLM on every run.

### Agent memory store (separate, not Alembic-managed)

Durable per-session memory is stored by LangGraph's `SqliteSaver` in its **own SQLite file** (path from `DATAANALYSIS_CHECKPOINT_DB`), keyed by `thread_id = session_id`. Its tables are created and owned by `langgraph-checkpoint-sqlite` — they are **not** part of the SQLAlchemy models or Alembic migrations, and should not be mixed into the metadata DB. Deleting a session closes its MCP pool; the checkpoint rows for that thread may be left orphaned or cleaned best-effort.

## Entities

### Entity: Dataset (`data_sources` / `DataSourceRow`)

Represents a connected **dataset** addressed by a URI. The existing `data_sources` table / `DataSourceRow` class is **reused** as the Dataset row (the table id is internal; renaming it would churn the `SessionDataSource` FK and many call sites for no user benefit). `name` is the **dataset name** (also the logical database name and the LLM tool canonical name); `type` is the **dataset type**.

A `parquet` dataset is a **directory of related Parquet files** — one uploaded CSV → one Parquet file → one table. A `postgresql` dataset (BETA, flag-gated) is an **external database**; its tables are introspected from `information_schema`. One MCP server is built **per dataset**, exposing **one capability per table**.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key (the dataset id; also keys the on-disk directory) |
| name | TEXT | yes | Dataset name = logical DB name = LLM tool canonical name (uniqueness enforced at the API layer, not a DB constraint) |
| type | TEXT | yes | Dataset type: `parquet` (internal) or `postgresql` (external, BETA). MySQL / DocumentDB deferred |
| description | TEXT | no | Optional user-provided description |
| uri | TEXT | no | Dataset URI. Internal: `parquet:///{name}` (model default-derives when empty). External: `postgresql://user:pass@host:port/db` (credentials stored **plaintext** for v0.1 BETA; never logged/displayed — always rendered via `DatasetURI.display()`) |
| tool_description | TEXT | no | LLM-generated dataset-level description, used as the MCP server's tool description |
| last_synced_at | TIMESTAMP | no | When the dataset's descriptions / external schema were last (re)generated by Sync |
| connection_error | TEXT | no | Last sanitized connection-check error (never contains credentials); null when healthy |
| created_at | TIMESTAMP | yes | When the dataset was connected |
| file_path | TEXT | no | **DEPRECATED** — kept nullable for rollback |
| parquet_path | TEXT | no | **DEPRECATED** — per-table now; moved to `DatasetTableRow`. Kept nullable for rollback |
| row_count | INTEGER | no | **DEPRECATED** — per-table now. Kept nullable for rollback |
| column_names_json | TEXT | no | **DEPRECATED** — per-table now. Kept nullable for rollback |
| schema_json | TEXT | no | **DEPRECATED** — per-table now. Kept nullable for rollback |
| capability_description | TEXT | no | **DEPRECATED** — per-table now; moved to `DatasetTableRow`. Kept nullable for rollback |

> Physical Parquet files live at `{datasets_dir}/{dataset_id}/{table}.parquet` — keyed by **dataset_id**, so renames never move files and same-named datasets never collide. There is **no** filesystem path in the URI.

### Entity: DatasetTable (`dataset_tables` / `DatasetTableRow`)

Child of Dataset — **one row per table** within a dataset (one Parquet file for `parquet`; one introspected table for `postgresql`). The dataset's MCP server exposes one **capability** per table.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key |
| dataset_id | TEXT (FK → Dataset.id) | yes | Owning dataset (indexed) |
| table_name | TEXT | yes | SQL table name = capability name; `sql_table_name(filename)`, auto-suffixed (`_2`, `_3`) on collision within the dataset |
| source_filename | TEXT | no | Original uploaded filename (parquet); null/source name for external |
| parquet_path | TEXT | no | Absolute path to the table's Parquet file (`{datasets_dir}/{dataset_id}/{table}.parquet`); **NULL for external** tables |
| row_count | INTEGER | no | Number of data rows (set after ingestion / introspection) |
| column_names_json | TEXT | no | JSON-encoded list of column names |
| schema_json | TEXT | no | JSON array `[{"name","dtype","nullable"}]` |
| capability_description | TEXT | no | LLM-generated description of this table's `query_{table}` capability, shown to the LLM |
| created_at | TIMESTAMP | yes | When the table was added |

Constraint: `UNIQUE(dataset_id, table_name)`.

> The MCP **capabilities** for a dataset are derived at runtime: one `query_{table}` tool per table, all registered on the dataset's single server (FastMCP requires unique tool names per server). All of a dataset's tables are views in **one** DuckDB connection, so any capability may JOIN the other tables in the **same** dataset; cross-**dataset** joins are not possible (separate servers) — the agent composes those across ReAct iterations. The LLM call is two-level: `{"tool":"<dataset_name>","capability":"<table_name>","arguments":{"query":"SELECT …"}}`. `inputSchema` is auto-generated by FastMCP from the `query(query: str)` signature; description = the table's `capability_description`.

### Entity: Session

A named conversation session. A session may span **multiple Datasets** — the agent connects one MCP server per attached dataset and sees all their capabilities at once. (Cross-**dataset** SQL joins in a single query are out of scope for v0.1: each dataset has its own server; the agent combines results across tool calls. Joins **within** a single dataset's tables are supported — they are views in one DuckDB connection.)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key |
| name | TEXT | no | Optional user-given name; defaults to "Session YYYY-MM-DD HH:MM" |
| created_at | TIMESTAMP | yes | Session start time |
| updated_at | TIMESTAMP | yes | Last activity time |

### Entity: SessionDataSource (join table)

Links Sessions to Datasets (many-to-many). Table/class names unchanged (`data_source_id` still references the dataset row).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| session_id | TEXT (FK → Session.id) | yes | Composite PK |
| data_source_id | TEXT (FK → Dataset.id) | yes | Composite PK (references `data_sources.id`) |

## Relationships

```
Dataset (M) ──< SessionDataSource >── (N) Session
Dataset (1) ──< DatasetTable (N)
Session (1) ──< QueryRecord (N)
QueryRecord (1) ──< AgentRun (N)
```

A session attaches **many** datasets; each dataset owns **many** tables. (There are no `Tool` / `ToolCapability` tables — removed when the tool layer became MCP. A dataset's capabilities are served by its per-dataset MCP server at runtime, one per table.)

## Bootstrap: CSV Upload

Upload requires a **dataset_name** + **dataset_type** (default `parquet`) and a CSV. A `Dataset` row plus its **first** `DatasetTable` child are created atomically:

1. The CSV is streamed to `{datasets_dir}/{dataset_id}/{table}.parquet`; `row_count`, `column_names_json`, and `schema_json` are extracted during conversion and stored on the **table** row.
2. The dataset-level `tool_description` and the per-table `capability_description` are generated by the LLM from a sample of **all** the dataset's tables (deterministic templates on failure, so uploads never fail).
3. `connection_check()` runs **before commit**; a broken dataset is never persisted (rollback on failure).
4. No tool rows are written — the dataset's MCP server materializes one `query_{table}` capability per table at query time.

**Adding more CSVs.** An existing dataset accepts more CSVs via `POST /datasources/{id}/add-csv`; **each new CSV = a new table** (`DatasetTable` child), with the table name auto-suffixed on collision within the dataset. Sync regenerates the dataset `tool_description` and every per-table `capability_description` from the schema of all tables, sets `last_synced_at`, and connection-checks before commit.

## Data Lifecycle

- **Dataset**: created on upload (with its first table); grows a `DatasetTable` per add-csv; deleted via explicit user action — `shutil.rmtree` of the **whole dataset directory** (`{datasets_dir}/{dataset_id}/`), plus its `DatasetTable` child rows, SessionDataSource links, and QueryRecord/AgentRun via the session.
- **DatasetTable**: created on upload / add-csv / external introspection; deleted with its parent dataset.
- **MCP pool**: per session — built lazily on the first query (`SessionPoolManager`), reused across queries, idle/LRU-evicted, and closed on session delete + app shutdown. Never persisted (rebuilt from each table's `parquet_path` / the dataset `uri` on demand); one server per dataset, one capability per table; its `ClientSession`s are transient per node. `close(session_id)` is lock-safe (acquires the session lock before teardown) so add-csv / sync / delete never close a DuckDB connection mid-query.
- **Session**: created when the user starts a session; links removed when a dataset is deleted.
- **QueryRecord**: created when the user submits a question; answer written after the pipeline completes; deleted with the Session.
- **AgentRun**: created at pipeline start; updated to `completed` or `failed` at end; deleted with the QueryRecord.

## Migration note

The dataset migration (new Alembic revision, `down_revision='b8e1f0a2c3d4'`, batch mode for SQLite) adds `data_sources.uri`, `last_synced_at`, and `connection_error`; creates the `dataset_tables` table with its `UNIQUE(dataset_id, table_name)` constraint; and **back-fills one `DatasetTable` child per legacy `data_sources` row** via a Python data migration (`op.get_bind()`): `table_name = sql_table_name(name)`, `parquet_path` **preserved at the row's existing path** (no file moves), copying `row_count` / `column_names_json` / `schema_json` / `capability_description`; the dataset row gets `uri = parquet:///{url-encoded name}` and `type` `csv` → `parquet`. The deprecated per-source columns are **kept** (nullable) for safe rollback. `downgrade` drops `dataset_tables` and the three new columns and restores `type = 'csv'` (lossless — legacy columns untouched). The pool defensively treats a dataset with zero children as legacy, synthesizing one table from the row's old `parquet_path`.

## Sensitive Data

- Uploaded files may contain PII depending on what the user provides. No special handling in v0.1 — Parquet files are stored as-is on local disk. Multi-user deployments must add access controls before exposing this app.
- No API keys stored in the database.
- External (`postgresql`, BETA) dataset credentials are stored **plaintext** in `data_sources.uri` for v0.1. They are **never** logged or displayed — every rendering goes through `DatasetURI.display()`, which strips credentials. A read-only `ATTACH` lets the agent read **any** table the credentials can reach (not only introspected ones), so the DB user must be scoped. External datasets are gated by `DATAANALYSIS_ENABLE_EXTERNAL_DATASETS` (default off; create path returns 501 when off).
