# Data Model

## Storage Technology

**SQLite** via SQLAlchemy 2.0 (sync) stores all **application metadata** (data sources, sessions, query history). Alembic manages schema migrations. The **data itself** lives in **Parquet** files on disk and is queried by DuckDB inside the per-source MCP server — it is never copied into SQLite.

A data source's tools are **not** stored as rows. They are exposed at runtime by the source's MCP server and discovered via the MCP client (`list_tools`). The only tool-related data persisted is the LLM-generated **descriptions**, kept on the `DataSource` so the server's tool description survives without re-calling the LLM on every run.

## Entities

### Entity: DataSource

Represents a connected data source. For v0.1 the only supported type is `csv` (the uploader also accepts `.xlsx`/`.json`, all normalized to Parquet).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key |
| name | TEXT | yes | User-facing name (defaults to filename) |
| type | TEXT | yes | Enum: `csv` (future: `api`, `graphql`, `shell`) |
| description | TEXT | no | Optional user-provided description |
| file_path | TEXT | no | Original on-disk path (legacy; uploads now stream straight to Parquet) |
| parquet_path | TEXT | no | Absolute path to the converted Parquet file — what the MCP server wraps |
| row_count | INTEGER | no | Number of data rows (set after ingestion) |
| column_names_json | TEXT | no | JSON-encoded list of column names |
| schema_json | TEXT | no | JSON array `[{"name","dtype","nullable"}]` |
| tool_description | TEXT | no | LLM-generated description of the dataset, used as the MCP server's tool description |
| capability_description | TEXT | no | LLM-generated description of the `run_query` tool, shown to the LLM |
| created_at | TIMESTAMP | yes | When the data source was connected |

> The MCP **tool** for a source is derived at runtime: tool key `<table_name>__run_query`, where `table_name = sql_table_name(name)`; `inputSchema` is auto-generated by FastMCP from the `run_query(query: str)` signature; description = `capability_description`.

---

### Entity: Session

A named conversation session. A session may span **multiple DataSources** — the agent connects one MCP server per attached source and sees all their tools at once. (Cross-source SQL joins in a single query are out of scope for v0.1: each server wraps one Parquet; the agent combines results across tool calls.)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key |
| name | TEXT | no | Optional user-given name; defaults to "Session YYYY-MM-DD HH:MM" |
| created_at | TIMESTAMP | yes | Session start time |
| updated_at | TIMESTAMP | yes | Last activity time |

### Entity: SessionDataSource (join table)

Links Sessions to DataSources (many-to-many).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| session_id | TEXT (FK → Session.id) | yes | Composite PK |
| data_source_id | TEXT (FK → DataSource.id) | yes | Composite PK |

---

### Entity: QueryRecord

One natural language question submitted by the user within a Session.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key |
| session_id | TEXT (FK → Session.id) | yes | Owning session |
| question | TEXT | yes | The user's natural language question |
| answer | TEXT | no | The LLM's final plain-text answer (null while processing) |
| status | TEXT | yes | `pending` / `completed` / `failed` |
| error_message | TEXT | no | Error detail if status=failed |
| iteration_count | INTEGER | no | Number of tool calls made to reach the answer |
| query_history_json | TEXT | no | JSON array of `{"tool": str, "arguments": dict, "result": str, "is_error": bool}` entries — the full tool-call trace |
| input_tokens | INTEGER | no | Total input tokens across all LLM calls |
| output_tokens | INTEGER | no | Total output tokens across all LLM calls |
| total_tokens | INTEGER | no | Sum of input + output tokens |
| estimated_cost_usd | REAL | no | Estimated API cost in USD |
| api_request_count | INTEGER | no | Number of LLM API calls made |
| created_at | TIMESTAMP | yes | When the query was submitted |
| updated_at | TIMESTAMP | yes | When the record was last modified |

---

### Entity: AgentRun

Internal record tracking each LangGraph pipeline invocation. One per QueryRecord (one-to-one in practice).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | TEXT (UUID) | yes | Primary key |
| query_record_id | TEXT (FK → QueryRecord.id) | yes | The query this run serves |
| status | TEXT | yes | `pending` / `completed` / `failed` |
| error_message | TEXT | no | Error detail if failed |
| created_at | TIMESTAMP | yes | Run start time |
| updated_at | TIMESTAMP | yes | Last update |

## Relationships

```
DataSource (M) ──< SessionDataSource >── (N) Session
Session (1) ──< QueryRecord (N)
QueryRecord (1) ──< AgentRun (N)
```

(There are no `Tool` / `ToolCapability` tables — removed when the tool layer became MCP. Tools are served by the per-source MCP server at runtime.)

## Bootstrap: CSV Upload

When a file is uploaded, a single `DataSource` is created atomically:

1. The file is streamed to `parquet_path`; `row_count`, `column_names_json`, and `schema_json` are extracted during conversion.
2. `tool_description` and `capability_description` are generated by the LLM from a sample of the Parquet (deterministic templates on failure, so uploads never fail).
3. No tool rows are written — the source's MCP server materializes its `run_query` tool at query time.

## Data Lifecycle

- **DataSource**: created on upload; deleted via explicit user action (cascades to SessionDataSource links, QueryRecord/AgentRun via the session, and the Parquet file on disk).
- **MCP server/session**: ephemeral — built in `load_data` and closed in `finalize`/`handle_error`. Never persisted.
- **Session**: created when the user starts a session; links removed when a source is deleted.
- **QueryRecord**: created when the user submits a question; answer written after the pipeline completes; deleted with the Session.
- **AgentRun**: created at pipeline start; updated to `completed` or `failed` at end; deleted with the QueryRecord.

## Migration note

The migration that introduces this model adds `data_sources.tool_description` and `data_sources.capability_description`, **back-fills them from the old `tools` / `tool_capabilities` rows**, then drops both tables. `downgrade` recreates the tables and restores the descriptions.

## Sensitive Data

- Uploaded files may contain PII depending on what the user provides. No special handling in v0.1 — Parquet files are stored as-is on local disk. Multi-user deployments must add access controls before exposing this app.
- No API keys stored in the database.
