# Agent Graph

> **Mandatory pre-coding checklist** — answer all questions before writing any node code.
> If any are missing, raise a blocker. (See `spec/engineering/ai-agents.md` Section 10.)
>
> 1. **Action type:** What does the LLM generate? → A two-level MCP tool call naming a dataset and one of its tables: `{"tool": "<dataset>", "capability": "<table>", "arguments": {"query": "<SQL>"}}`
> 2. **Termination signal:** How does the LLM say it's done? → Response starts with `FINAL ANSWER:`
> 3. **Recoverable vs fatal errors:** DuckDB SQL errors and non-SELECT SQL → recoverable (the MCP tool returns `isError=True`, fed back to the LLM). Unknown tool name → recoverable. LLM-call failure, missing Parquet, MCP server/session failure → fatal.
> 4. **Max iterations default:** `10` (configurable via `DATAANALYSIS_MAX_AGENT_ITERATIONS`)
> 5. **In-session resource:** A **per-session** pool (`tools/mcp/pool.py`, the `SessionPoolManager`) holds one built MCP server (+ its DuckDB connection) per attached dataset for the session's lifetime. It is built lazily on the session's first query and reused by every later query. `ClientSession`s are **transient** — opened and closed within a single node. The manager (not the graph) owns teardown: on session delete, app shutdown, and idle/LRU eviction.
> 6. **Memory:** durable per-session memory via a LangGraph **`SqliteSaver`** checkpointer with `thread_id = session_id`. The checkpointed state carries a growing `conversation` (prior Q&A turns); per-query scratch is reset via the `ainvoke` input.
> 7. **Tools the agent has:** One MCP server **per dataset**, exposing **one capability per table**. The tool's canonical name = the dataset name; each capability's name = a table name, with `inputSchema` auto-generated by FastMCP and description = that table's `capability_description`. (Server-side the capability is registered as a `query_{table}` tool — FastMCP requires unique tool names per server — but the LLM addresses it as `capability: <table>`.) A capability may JOIN the other tables in the same dataset (all tables are views in one DuckDB connection). Discovered once when the session pool is built; `plan_action` reads them from the manager (not from state).
> 8. **Exact tool invocation format:** the two-level JSON above — `{"tool": "<dataset>", "capability": "<table>", "arguments": {"query": "<SQL>"}}`. The dataset namespaces the table, so two datasets can share a table name with no collision.
> 9. **What triggers tool registration:** dataset creation (upload a named dataset, or connect a database) → a Dataset row + one `DatasetTableRow` per table; the dataset's MCP server is built when the **session** pool is first acquired, with one view (and one capability) per table.

## State

The graph is checkpointed per session, so state is split into **durable memory** (accumulates across
queries, restored by the checkpointer) and **per-query scratch** (reset each query via the `ainvoke`
input). Tools/schema are **not** in state — they are read from the `SessionPoolManager` by `session_id`,
keeping checkpoints small.

```python
class AgentState(TypedDict, total=False):
    # Identity
    run_id: str
    query_record_id: str
    session_id: str          # also the checkpointer thread_id

    # Input (scratch — supplied fresh each query)
    question: str

    # Durable memory (reducer-appended; survives across queries via the checkpointer)
    conversation: Annotated[list[dict], _append_turns]   # [{"question": str, "answer": str}]

    # Per-query scratch (reset via the ainvoke input each query)
    action_history: list[dict]   # [{"tool": str, "capability": str, "arguments": dict, "result": str, "is_error": bool}]
    iteration_count: int
    llm_response: str
    answer: str
    input_tokens: int
    output_tokens: int
    total_tokens: int
    estimated_cost_usd: float | None
    api_request_count: int

    # Control
    error: str | None
```

> The MCP servers + DuckDB connections (not serialisable) live in the `SessionPoolManager` keyed by
> `session_id`, never in state. `ClientSession`s are transient — opened/closed within a single node.

## Async lifecycle (read before touching nodes)

LangGraph runs **each node in its own asyncio task** (verified empirically). AnyIO ties an async context manager's cancel scope to the task that entered it, so a `ClientSession` (an async CM) **cannot be opened in one node and closed in another**. The design therefore:

- All nodes are `async def`. `run_pipeline()` (sync, on the request's daemon thread) acquires the session's lock, then drives `asyncio.run(ainvoke(...))`.
- The **`SessionPoolManager`** (`tools/mcp/pool.py`, keyed by `session_id`) holds only **plain, task-safe objects** across nodes and across queries: one built `FastMCP` server (+ DuckDB connection) per dataset. It never holds open `ClientSession`s.
- Each MCP operation opens a **transient** in-memory `ClientSession` within a single node: `plan_action` reads cached tool descriptors from the manager (no session needed); `execute_action` opens a session to `call_tool()`.
- **Pool teardown is NOT in the query path.** `finalize`/`handle_error` do not close the pool — the manager closes it on session delete, app shutdown, and idle/LRU eviction.
- **Per-session serialization:** the pool's DuckDB connection is not safe for concurrent use, so `run_pipeline` holds a per-session `threading.Lock` for the whole query; eviction skips locked (in-use) sessions.
- **Do not** add parallel fan-out nodes or span an MCP `ClientSession` across nodes. The sync LLM `complete()` and SQLAlchemy calls are made directly inside the async nodes (the dedicated thread makes blocking harmless).

## Nodes

The per-query loop is **`plan_action → execute_action → finalize / handle_error`**. There is no
`load_data` node — datasets and their tables are loaded once when the session pool is built
(`SessionPoolManager.acquire(session_id)`, called from `run_pipeline` before the graph runs).

### `plan_action`  (entry point)

**Reads from state:** `question`, `conversation`, `action_history`, `iteration_count`, token fields, `session_id`

**Writes to state:** `llm_response`, `answer` (if FINAL ANSWER), token fields

**External calls:** OpenRouter (or stub) — sync `complete()`, called directly. Failure → fatal.

**Behaviour:** Reads the session's grouped tools from the `SessionPoolManager` (by `session_id`) — one entry per dataset, each with its `tool_description` and a list of capabilities (table name + description + columns + parameter schema). Builds a prompt containing the available datasets and their capabilities, the **conversation so far** (`conversation`, durable memory), the user's question, and the current-query `action_history`. Sends to the LLM; accumulates tokens. `FINAL ANSWER:` → set `answer`, route to `finalize`; else store `llm_response`, route to `execute_action`. Stub tag: `<node:plan_action>`.

### `execute_action`

**Reads from state:** `session_id`, `llm_response`, `action_history`, `iteration_count`

**Writes to state:** `action_history` (appended), `iteration_count` (incremented)

**External calls:** `await manager.call_tool(session_id, tool, capability, arguments)` → DuckDB SELECT. SQL/unknown-tool/unknown-capability = recoverable; missing pool = fatal.

**Behaviour:** Parse `{"tool","capability","arguments"}` (bad JSON or a missing `capability` → recoverable, loop back). Route through the manager (it resolves `(tool, capability)` to that dataset's server's `query_{capability}`, opens a transient session → `call_tool` → `CallToolResult.content[0].text` + `.isError`). Append the observation `{"tool", "capability", "arguments", "result", "is_error"}`, increment `iteration_count`; at the cap, set `error`.

### `finalize`

**Reads:** `query_record_id`, `question`, `answer`, `iteration_count`, `action_history`, token fields

**Writes:** returns the new `conversation` turn `{"question", "answer"}` (reducer-appended → durable memory).

**External calls:** SQLite — update QueryRecord (answer, iteration_count, query_history_json, tokens, status=completed) and AgentRun (completed). **Does not close the pool.**

### `handle_error`

**Reads:** `error`, `query_record_id`. SQLite — mark QueryRecord/AgentRun `failed` (best-effort). **Does not close the pool.** Terminates the graph.

## Graph Assembly

`build_graph()` returns an **uncompiled** `StateGraph` builder; `run_pipeline` compiles it per query
with the session's checkpointer so memory is bound to `thread_id = session_id`.

```python
def build_graph() -> StateGraph:
    g = StateGraph(AgentState)
    g.add_node("plan_action", plan_action)       # all async def
    g.add_node("execute_action", execute_action)
    g.add_node("finalize", finalize)
    g.add_node("handle_error", handle_error)
    g.set_entry_point("plan_action")
    g.add_conditional_edges("plan_action", route_after_plan,
        {"execute_action": "execute_action", "finalize": "finalize", "handle_error": "handle_error"})
    g.add_conditional_edges("execute_action", route_after_execute,
        {"plan_action": "plan_action", "handle_error": "handle_error"})
    g.add_conditional_edges("finalize", route_after_finalize, {"end": END, "handle_error": "handle_error"})
    g.add_edge("handle_error", END)
    return g

# per query, inside the run's event loop:
async with AsyncSqliteSaver.from_conn_string(settings.checkpoint_db) as saver:
    graph = build_graph().compile(checkpointer=saver)
    final = await graph.ainvoke(_fresh_input(state), config={"configurable": {"thread_id": session_id}})
```

`_fresh_input` supplies the new `question` and resets all scratch fields (`action_history=[]`,
`iteration_count=0`, `llm_response=""`, `answer=""`, usage=0, `error=None`); it omits `conversation`
so the reducer keeps the restored history. Routing functions stay **sync**.

```
  Description: 2024 sales records and the regional lookup that supports them.

  capability: regions  (table)
    Columns: region_id (BIGINT), region_name (VARCHAR)

SQL dialect: DuckDB. Only SELECT statements are permitted. Each capability's query runs

To finish:     FINAL ANSWER: <your complete answer here>
```

## Concurrency & Memory Model

- One query runs at a time **per session**, serialized by a per-session `threading.Lock` (the shared DuckDB connection is not concurrency-safe). Different sessions run independently.
- Each query owns one event loop (`asyncio.run`); the session pool (servers + DuckDB conns) persists across queries in the `SessionPoolManager`.
- **Memory:** durable via `AsyncSqliteSaver` keyed by `thread_id = session_id`; `conversation` accumulates and survives restarts. Only `conversation` is kept in the durable state; scratch is reset each query.
- **Pool lifecycle:** lazy build on first query; **idle/LRU eviction** (`max_session_pools`, `session_pool_idle_seconds`); closed on session delete and app shutdown; invalidated when a session's sources change.
