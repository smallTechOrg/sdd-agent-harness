# FR-001 — Data Analyst Agent

**Date:** 2026-06-22  
**Author:** researcher  
**Status:** done

---

## Problem Statement

Data analysts spend significant time writing SQL, switching between tools, and manually formatting results into presentations. A conversational data analyst agent should let users upload their own datasets, ask questions in plain English, and receive rich, structured answers (tables, charts, dashboards) — all from a single interface. The agent should behave like a senior data analyst: generating correct SQL, explaining its reasoning, and suggesting follow-up angles. Every query should be logged for auditability.

---

## Target Users

| User | Context | Primary need |
|------|---------|--------------|
| Business analyst | Has CSV/JSON exports from BI tools; SQL-literate but not expert | Ask cross-dataset questions without writing joins |
| Data scientist | Explores new datasets quickly before writing ETL | Fast NL→SQL iteration with chart output |
| Non-technical stakeholder | Receives data files; wants to explore without SQL | Plain-English answers with auto-generated charts |

## Success Criteria (EARS)

### Dataset management
- [ ] WHEN a user uploads a CSV or JSON file with a given name, the system SHALL register it as a named DuckDB table, persist it across server restarts, and confirm registration with row count and inferred schema.
- [ ] WHEN a user lists datasets, the system SHALL return all registered datasets with name, format, row count, column schema, and upload timestamp.
- [ ] WHEN a user deletes a dataset by name, the system SHALL drop the DuckDB table and remove the metadata record.
- [ ] IF a user uploads a file with a name that already exists, THEN the system SHALL replace the existing dataset and log the replacement in the audit log.

### Natural language querying
- [ ] WHEN a user sends a natural language question referencing one or more dataset names, the system SHALL generate a valid DuckDB SQL query, execute it, and return a structured result within 30 seconds.
- [ ] WHEN the LLM generates SQL, the system SHALL display the SQL alongside the result so the user can inspect it.
- [ ] IF the generated SQL fails to execute, THEN the system SHALL retry with the error message as feedback (up to 2 retries) and return a plain-English explanation if all retries fail.
- [ ] WHEN a user asks a cross-dataset question, the system SHALL generate a JOIN query across the relevant DuckDB tables without manual schema hints.

### Rich responses
- [ ] WHEN a query result has tabular data (≤ 10k rows returned), the system SHALL render it as a formatted HTML table in the UI with column headers and pagination (50 rows/page).
- [ ] WHEN the LLM determines a result is best shown as a chart, the system SHALL render a bar, line, scatter, or pie chart (auto-selected) using Plotly in the Next.js UI.
- [ ] WHEN a query returns a scalar summary (e.g. "total revenue"), the system SHALL render it as a large formatted number with a one-sentence plain-English interpretation.
- [ ] WHEN a query returns a text explanation, the system SHALL render it as formatted markdown.

### Senior analyst workflow
- [ ] WHEN the system returns any query result, it SHALL suggest 2–3 follow-up questions relevant to the result.
- [ ] The system SHALL accept follow-up questions in the same session that reference the prior query's context (e.g. "now filter by region = 'West'").
- [ ] WHEN a user asks "explain this", the system SHALL provide a plain-English explanation of the SQL it generated and why.

### Dashboard
- [ ] WHEN a user pins a query result, the system SHALL add it as a panel in the dashboard view, persisted to the session.
- [ ] The system SHALL render all pinned panels in a grid layout on the dashboard page with their chart or table.
- [ ] WHEN a user removes a pinned panel, the system SHALL update the dashboard immediately.

### Token economy
- [ ] The system SHALL send only column names and types (not sample data) in the schema context prompt, unless the user explicitly requests samples. [ASSUMPTION: schema-only prompting is sufficient; sample data can be added via follow-up]
- [ ] The system SHALL cache the schema context per session so it is not re-sent on every query turn.

### Persistent sessions
- [ ] The system SHALL assign a unique session ID per browser tab and persist it in localStorage with a Math.random fallback for non-secure contexts.
- [ ] WHILE a session is active, the system SHALL maintain the full query history (NL question, SQL generated, result summary, timestamp) queryable via API.
- [ ] The system SHALL allow a user to load a past session by session ID and replay its query history.

### Audit logging
- [ ] The system SHALL log every SQL execution to an `audit_log` table: session_id, query_text, dataset_names (array), rows_returned, duration_ms, timestamp, success (bool), error_message (nullable).
- [ ] WHEN a user calls `GET /audit-log`, the system SHALL return the last 100 audit entries filterable by session_id and date range.
- [ ] IF the LLM provider is set to `stub`, THEN the system SHALL pass all unit tests with no API key and no network call, and the UI SHALL display a prominent stub-mode banner.

## Key Constraints

| Constraint | Value | Reason |
|-----------|-------|--------|
| Stack | Python 3.12 + FastAPI + LangGraph + DuckDB + SQLite | Analytics workload (DuckDB columnar); agent loop (LangGraph); metadata/sessions/audit (SQLite spine) |
| Database | DuckDB (datasets) + SQLite (metadata, sessions, audit_log) | Per C-DUCKDB-RECIPE: analytics needs DuckDB, not SQLite-adapted relational recipe |
| LLM provider | Gemini 2.5 Flash (`google-genai` SDK per C-LLM-SDK) configurable via `ANALYST_LLM_PROVIDER=gemini\|stub` | User-selected for lower per-token cost; token-economical; schema-only prompting; provider-agnostic interface |
| Frontend | Next.js + Plotly (SSR-safe via dynamic import per C-PLOTLY-SSR) + react-markdown | Charts + dashboard require UI; markdown rendering per C-MD-RENDER |
| Recipes | `python-fastapi-duckdb` + `frontend-nextjs` | Both needed |
| Deploy target | Local demo (port 8001 backend, 3000 frontend) | Local-first; Render on request only |
| API keys needed | `GEMINI_API_KEY` | Required for LLM step; stub mode runs without it |
| Offline gate | Full test suite passes with `ANALYST_LLM_PROVIDER=stub`, no key, no network | Non-negotiable #3 per C-STUB |
| Session ID | Per-tab UUID via `crypto.randomUUID()` in `useEffect` + localStorage | Per C-SESSION-SCOPE + C-SSR-BROWSER-API |
| Port | 8001 (backend), 3000 (frontend) | Per C-PORT-8001 |
