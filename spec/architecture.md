# Architecture

## System Overview

A personal data analysis agent running entirely on the local machine. The user interacts through a Next.js chat UI served by FastAPI at `http://localhost:8001/app/`. Uploaded files (CSV/Excel) are stored on the local filesystem; a SQLite database holds app metadata (file registry, schema cache, run history). When the user asks a question, a LangGraph graph generates Pandas code, executes it locally in a sandboxed `exec()` environment, and sends only a small result sample (≤500 rows) to the Gemini LLM for reasoning. The LLM returns a text answer and a Plotly.js chart spec that the frontend renders interactively inline. PostgreSQL (Phase 2) is an additional external data source queried via SQL generated by the same graph.

## Layers

| Layer | Responsibility |
|-------|----------------|
| Frontend (Next.js static export) | Chat UI, file upload, Plotly chart rendering, sidebar |
| API (FastAPI) | HTTP routing, multipart upload handling, static file serving |
| Agent (LangGraph) | Orchestrates the 4-node analysis graph; manages state |
| Code execution (local sandbox) | Runs LLM-generated Pandas code in a restricted exec() environment |
| LLM (Gemini) | Generates analysis code/SQL (plan node) and text answers + chart specs (reason node) |
| Storage — App DB (SQLite) | File registry, schema cache, run history |
| Storage — Files (local FS) | Raw uploaded CSV/Excel files at `data/uploads/<uuid>/` |
| Storage — External DB (PostgreSQL) | User's existing data source; queried at analysis time only (Phase 2) |

## Data Flow

```
1. UPLOAD
   User drops CSV → POST /api/files/upload (multipart)
   → FastAPI writes file to data/uploads/<uuid>/<filename>
   → Pandas reads file, extracts schema (column names, dtypes, 3-row sample as JSON)
   → schema_json persisted to uploaded_files in SQLite
   → Response: {file_id, filename, schema_preview: {columns, dtypes, sample_rows}}

2. QUESTION
   User types question → POST /api/analysis/run {file_id, question, session_id?}
   → FastAPI calls run_agent(file_id, question, session_id)
   → LangGraph invokes the graph:

   a. load_dataset
      - Looks up file_path from uploaded_files (SQLite by file_id)
      - Loads CSV/Excel into pandas DataFrame in-memory
      - Attaches df + schema_info to AgentState
      - (Phase 2) If source_type=postgres: opens psycopg2 connection via POSTGRES_DSN,
        introspects table schema, stores connection handle in state

   b. plan_analysis  [LLM call: gemini-2.5-flash]
      - Input: schema_info (column names, dtypes, 3-row sample) + question
        + (Phase 3) conversation_history of prior turns
      - Output: structured JSON {code_type: "pandas"|"sql", code: "..."}
      - Pandas code constraints enforced via prompt; code is validated
        (AST-check for forbidden imports) before execution

   c. execute_code  [local exec — no LLM]
      - Runs Pandas code in restricted exec() sandbox (df in scope,
        forbidden builtins removed, 30-second timeout)
      - OR executes SQL via psycopg2 against PostgreSQL (Phase 2)
      - Captures result (DataFrame or scalar); trims to max 500 rows
      - On exception → routes to handle_error

   d. reason_answer  [LLM call: gemini-2.5-pro]
      - Input: original question + result sample (≤500 rows as CSV string)
        + schema_info + column dtypes
      - Output: JSON {answer_text: "...", chart_spec: {chart_type, data, layout}}

   e. finalize
      - Persists AnalysisRun row (answer_text, chart_spec_json, status="completed")
      - Returns final state with answer_text + chart_spec

3. RENDER
   Frontend receives {run_id, answer, chart_spec} from POST /api/analysis/run
   → AnswerCard renders text answer in markdown
   → ChartPanel calls Plotly.newPlot(divId, chart_spec.data, chart_spec.layout,
     {responsive: true})
   → Chart rendered interactively inline (zoom, hover, pan)
```

## Chart Pipeline

The `reason_answer` node emits a `chart_spec` JSON object with this exact shape:

```json
{
  "chart_type": "bar",
  "data": [
    {
      "type": "bar",
      "x": ["A", "B", "C"],
      "y": [10, 20, 15],
      "name": "Count"
    }
  ],
  "layout": {
    "title": "Top values by count",
    "xaxis": {"title": "Category"},
    "yaxis": {"title": "Count"}
  }
}
```

The frontend passes `chart_spec.data` and `chart_spec.layout` directly to `Plotly.newPlot(divId, data, layout, {responsive: true})`. The `chart_type` field is informational only; the `type` key inside each trace object controls Plotly rendering. If `chart_spec` is null or missing, the AnswerCard renders text only.

**Chart type selection rules** (enforced by `reason_answer` system prompt):
- `bar` — categorical comparisons, top-N rankings, counts by group
- `line` — time-series data, trends over an ordered sequence
- `scatter` — correlations between two continuous variables
- `pie` — part-of-whole proportions with ≤6 categories; otherwise `bar`

## Security Notes

- **No raw data to LLM.** Only aggregated results (≤500 rows) or scalar values are included in LLM prompts. Full dataset rows are never serialized into a prompt.
- **Local code execution sandbox.** The `execute_code` node runs LLM-generated Pandas code inside `exec()` with a restricted globals dict. Forbidden: `import os`, `import sys`, `import subprocess`, `open()`, `socket`, `requests`, `urllib`. The `__builtins__` dict is replaced with a whitelist. The generated code is also statically checked with Python's `ast` module for forbidden import statements before execution. Timeout: 30 seconds.
- **File storage.** Uploaded files are stored on the local filesystem under `data/uploads/<uuid>/`. No cloud upload. Files are referenced by UUID-keyed path stored in SQLite.
- **PostgreSQL DSN.** Provided via `POSTGRES_DSN` in `.env`; never committed to source control or stored in SQLite plaintext.

## Frontend Architecture

- **Next.js 15 static export** (`output: 'export'`, `basePath: '/app'`, `trailingSlash: true`).
- Built with `cd frontend && pnpm build` → output at `frontend/out/`.
- FastAPI mounts `frontend/out/` at `/app` via `StaticFiles(html=True)`.
- **Single-origin run path:** `http://localhost:8001/app/` — the gate and test handoff always use this path.
- All API calls from the frontend use relative paths prefixed with `/api/` (e.g., `fetch('/api/files/upload', ...)`).
- Plotly.js loaded via `pnpm add plotly.js-dist-min` (minified bundle) and imported as `import Plotly from 'plotly.js-dist-min'`.
- `NODE_OPTIONS=--no-experimental-webstorage` is set in all `package.json` scripts to prevent `localStorage` crash on Node ≥25.

## External Dependencies

| Dependency | Purpose | Failure Mode |
|------------|---------|--------------|
| Google Gemini API | Code generation (plan node) and answer + chart-spec reasoning (reason node) | LLM call fails → `handle_error` returns user-facing message; run status set to "failed" |
| Local filesystem | Storage for uploaded CSV/Excel files | Write fails → 500 returned to upload endpoint with clear message |
| SQLite (data/agent.db) | App metadata: file registry, schema cache, run history | DB unreachable → FastAPI startup fails with clear log message |
| PostgreSQL (Phase 2) | External data source queried at analysis time | Connection fails → `handle_error` with "Cannot connect to PostgreSQL" message; app continues |
| LangSmith (optional) | Distributed tracing of LangGraph runs | Tracing failure is non-fatal; run continues without trace |

## Deployment Model

Local-only. Single process: `uv run python -m src` starts uvicorn on port 8001. The frontend is a static export pre-built with `cd frontend && pnpm build`. No cloud deployment, no containerization required. Data persists in `data/agent.db` (SQLite) and `data/uploads/` (filesystem) between server restarts.

## Stack

| App database | SQLite via SQLAlchemy 2.0 (existing skeleton; `data/agent.db`) |

| Data manipulation | pandas ≥2.2, openpyxl ≥3.1 (Phase 3 Excel) |
| Frontend framework | Next.js 15 + React 19 |
| Frontend styling | Tailwind CSS v4 (existing skeleton) |
| Charting | `plotly.js-dist-min` (pnpm) |
| Dependency management | uv (Python), pnpm (frontend) |
| Structured logging | structlog ≥24.1 (already in `pyproject.toml`) |
| Observability / tracing | LangSmith via `langchain-core`; env vars `LANGCHAIN_TRACING_V2`, `LANGCHAIN_API_KEY` |
| Migrations | Alembic ≥1.13 (already in `pyproject.toml`) |
| Testing — backend | pytest ≥8.2, pytest-asyncio ≥0.23, httpx ≥0.27 |
| Testing — E2E | Playwright (`@playwright/test`; chromium only; `tests/e2e/`) |
| Dev port | 8001 |

> **Assumed:** `pandas` and `openpyxl` are added to `[project.dependencies]` by the code-generator (not currently in `pyproject.toml`).

> **Assumed:** `psycopg2-binary` is added to `[project.dependencies]` (not dev-only) so Alembic and Phase 2 queries work in all environments.

> **Assumed:** `langchain-core` is added to `[project.dependencies]` for LangSmith tracing. Tracing is opt-in: active when both `LANGCHAIN_TRACING_V2=true` and `LANGCHAIN_API_KEY` are set; silent no-op otherwise.

> **Assumed:** `plotly.js-dist-min` is installed via `pnpm add plotly.js-dist-min` in the frontend package.

> **Assumed:** `gemini-2.5-flash` and `gemini-2.5-pro` are the verified current model identifiers for the Gemini API as of 2026. If the provider deprecates a name, the env-var override (`AGENT_LLM_MODEL_PLAN`, `AGENT_LLM_MODEL_REASON`) allows correction without a code deployment.

**Avoid:**
- SQLite as a substitute for PostgreSQL in Phase 2+ tests — PostgreSQL tests must run against PostgreSQL.
- Sending full dataset rows to the LLM — only aggregated results (≤500 rows).
- `pnpm dev` / `localhost:3000` as the gate or test path — single-origin `localhost:8001/app/` is the only valid test path.
- Storing `POSTGRES_DSN` in plaintext in the SQLite app DB.
